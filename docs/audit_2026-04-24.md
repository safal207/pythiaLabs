# Глубокий аудит проекта `liminal-pythia` (на 2026-04-24)

## 1) Executive summary

Проект выглядит как **рабочий MVP-скелет** для цикла refine/planner с двумя вычислительными путями (NIF + fallback) и отдельным Rust worker через Port. По коду видно хорошее направление (прозрачный trace, простая архитектура, акцент на безопасной изоляции через Port), но до продуктового уровня сейчас есть заметный разрыв в **надёжности, тестировании, операционности, API-границах и безопасности входных данных**.

**Оценка готовности (субъективно):**
- Технический фундамент MVP: **6/10**
- Готовность к прод-эксплуатации: **3/10**
- Готовность к продуктовой итерации с внешними пользователями: **4/10**

---

## 2) Что уже хорошо

1. **Чистое разделение ответственности**:
   - Planner (управление шагами), Executor (применение операций/оценка), Kernels (ускорение), Critic (hook) — логично и расширяемо.
2. **Прозрачность вычислений**:
   - Ведётся trace по шагам; это важно для explainability.
3. **Два режима производительности**:
   - Rust NIF + Elixir fallback для устойчивости.
4. **Изоляция тяжёлого solver-а через Port**:
   - Это лучше с точки зрения fault containment, чем всё делать в NIF.

---

## 3) Ключевые технические риски

### P0 (блокеры перед продуктом)

1. **Нет тестового покрытия (unit/integration/property)**
   - Невозможно гарантировать корректность редких и граничных случаев.
   - Нет автоматической валидации регрессий перед релизом.

2. **Проблема в fallback Levenshtein для пустых строк**
   - В `KernelsFallback.do_lev/2` используется `Enum.reduce(1..m, ...)`. Для `m == 0` диапазон `1..0` в Elixir не пустой (идёт в обратном направлении), что приводит к неправильной логике и потенциально некорректному результату.
   - Это риск неверного score и, следовательно, неверного поведения Planner.

3. **Дублирование модуля `Mix.Tasks.Compile.RustlerFallback`**
   - Есть две реализации одного и того же модуля в разных путях (`lib/...` и `mix/tasks/...`).
   - Это может создавать неоднозначность компиляции/загрузки и усложнять сопровождение.

### P1 (сильно влияет на стабильность)

4. **Port worker не валидирует вход**
   - В Rust worker есть `unwrap()` при чтении/парсинге JSON и предположения о форме `grid`.
   - Любой невалидный запрос может аварийно завершить процесс worker-а.

5. **Planner/Critic контракт пока заглушка**
   - Critic не влияет на поведение при `no_improve`, хотя интерфейс уже заложен.
   - Продуктово это пока «пустой» крючок без measurable impact.

6. **Нет наблюдаемости (telemetry/метрики/корреляция шагов)**
   - Trace есть в памяти, но нет системной телеметрии, событий ошибок, latency-метрик.

### P2 (важно для масштабирования)

7. **Нет публичного API-слоя (HTTP/gRPC) и версионирования контрактов**
   - Сейчас использовать можно только как библиотеку/скрипты.

8. **Нет явной модели конфигурации окружений**
   - `config/config.exs` минимален; нет runtime-конфигов, feature flags, ограничителей.

9. **Нет формализованной модели безопасности**
   - Не определены лимиты входов (размер строки, размер лабиринта, таймауты, memory ceilings).

---

## 4) Продуктовые гэпы (до «первых пользователей»)

1. **Отсутствует чёткий пользовательский сценарий v1**
   - Есть демонстрации, но нет зафиксированного JTBD/ICP/acceptance criteria.
2. **Нет SLA/SLO и operational target-ов**
   - Нельзя определить, что релиз «достаточно хорош».
3. **Нет UX вокруг explainability**
   - Trace есть технически, но нет нормализованного формата объяснения для пользователя/клиента.

## 6) Definition of Done перед внешним пилотом

Минимум перед пилотом:

1. 80%+ покрытие критичных модулей (`Planner`, `Executor`, `KernelsFallback`).
2. Ноль известных P0 багов.
3. Worker устойчив к невалидному вводу (без паник/крашей).
4. Базовая телеметрия и логи доступны.
5. Документированный API v1 + примеры запросов/ответов.
6. Набор golden test cases проходит в CI.
