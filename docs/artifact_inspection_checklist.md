# Artifact Inspection Checklist

Status: reviewer-facing checklist.

This checklist helps reviewers inspect generated PythiaLabs evidence artifacts without trusting terminal output alone.

Primary references:

- [`../examples/paid_review_demo_expected_output.md`](../examples/paid_review_demo_expected_output.md)
- [`paid_review_demo_reviewer_checklist.md`](paid_review_demo_reviewer_checklist.md)
- [`evidence_artifact_schema.md`](evidence_artifact_schema.md)
- [`paid_review_demo_sample_reviewer_report.md`](paid_review_demo_sample_reviewer_report.md)

## Before running

Confirm you are reviewing a local deterministic demo, not a production deployment.

PythiaLabs currently does not claim:

- production security certification;
- regulatory compliance certification;
- wallet protection;
- smart-contract auditing;
- on-chain enforcement;
- complete agent safety.

See [`NON_CLAIMS.md`](NON_CLAIMS.md).

## Run command

From the repository root:

```bash
make demo
```

Equivalent direct command, when available:

```bash
mix run examples/paid_review_demo.exs
```

## Artifact location

The paid review demo writes the artifact bundle to:

```text
examples/output/paid_review_demo_artifact.json
```

The file is regenerated by the demo and should not be treated as a manually curated fixture.

## Terminal-output checks

Before inspecting the artifact bundle, confirm the terminal output says:

- `PythiaLabs — Paid Review Demo`;
- `scenario_set=paid_review_demo_2026_05`;
- four scenarios were evaluated;
- all observed outcomes matched expected outcomes;
- each scenario reported evidence verification;
- the counterfactual produced a decision flip;
- final result is `PASS`.

## Artifact-level checks

Open:

```text
examples/output/paid_review_demo_artifact.json
```

Check the following:

- [ ] Bundle-level schema/version exists.
- [ ] Scenario set identifier exists.
- [ ] There are four scenario records.
- [ ] Scenario IDs match the expected set.
- [ ] Each scenario record has a decision/status field.
- [ ] Each scenario record has a stable stop reason.
- [ ] Each scenario record has evidence trace/check data.
- [ ] Each scenario record has a SHA-256 digest or digest field.
- [ ] Each scenario record reports verification status.
- [ ] The counterfactual section exists.
- [ ] The counterfactual changes the decision in the expected direction.

Expected scenario set:

| Scenario | Expected decision | Expected stop reason |
|---|---:|---|
| `accepted_transfer` | accepted | `treasury_transfer_accepted` |
| `quorum_not_met` | rejected | `quorum_not_met` |
| `timelock_not_satisfied` | rejected | `timelock_not_satisfied` |
| `transfer_window_expired` | rejected | `transfer_expired` |

## Evidence-trace checks

Across the scenario records, reviewers should see checks such as:

- `proposal_match_check`
- `permission_check`
- `quorum_check`
- `voting_window_check`
- `timelock_check`
- `authorization_valid_time_check`
- `authorization_transaction_time_check`
- `transfer_expiration_check`

A rejected scenario should show a trace/check corresponding to its stop reason.

## Digest checks

The demo uses SHA-256 digest fields over canonicalized evidence payloads.

Check:

- [ ] digest field exists;
- [ ] algorithm is SHA-256 or clearly documented demo SHA-256 mode;
- [ ] digest is not an all-zero placeholder in generated output;
- [ ] terminal output reports evidence verification;
- [ ] artifact record reports verification status.

Do not expect digest values to match this document. Runtime/data-dependent digests should differ.

## Counterfactual checks

The expected counterfactual changes:

```json
{"governance_record":{"quorum_met":true}}
```

Expected result:

```text
quorum_not_met: rejected -> accepted
```

This demonstrates that the deterministic demo decision changes when relevant evidence changes.

## Red flags

Treat these as review warnings:

- missing artifact bundle after `make demo`;
- fewer or more than four scenario records without documentation;
- missing stop reason;
- missing evidence trace/check list;
- missing digest field;
- unverifiable evidence;
- accepted/rejected outcomes not matching expected output;
- counterfactual missing or not changing the decision;
- artifact text claiming production security, compliance certification, wallet protection, or complete agent safety.

## Non-claims

Artifact verification in the current demo means:

```text
the generated evidence payload round-tripped through the demo verification path
```

It does not mean:

- production cryptographic assurance;
- legal admissibility;
- regulatory compliance;
- security certification;
- wallet safety;
- smart-contract correctness;
- complete protection against malicious agents.

## Reviewer conclusion template

```text
I ran `make demo`, inspected terminal output, and reviewed the generated artifact bundle.

Result:
- scenario count matched expectations;
- decisions and stop reasons matched expected outcomes;
- evidence traces were present;
- digest/verification fields were present;
- counterfactual behavior matched expectation;
- no production-security or compliance claims were introduced.

Conclusion: the artifact bundle is a useful deterministic review artifact for the PythiaLabs pre-execution evidence-gate concept.
```
