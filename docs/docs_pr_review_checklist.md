# Docs PR Review Checklist

Status: maintainer-facing checklist.

Use this checklist when reviewing documentation-only pull requests for PythiaLabs.

## Basic checks

- [ ] The PR references an issue or clearly explains why no issue is needed.
- [ ] The PR stays within the issue scope.
- [ ] The change is docs-only if the issue says docs-only.
- [ ] No site/build/runtime/demo/workflow files were changed unless explicitly requested.
- [ ] The PR summary explains what changed.
- [ ] The PR summary states what was not changed.

## Link checks

- [ ] New links use correct relative paths.
- [ ] Existing links were not broken.
- [ ] README/docs index links point to the intended files.
- [ ] External links are necessary and reviewer-relevant.

## Claim-safety checks

- [ ] Wording avoids production-security claims.
- [ ] Wording avoids compliance-certification claims.
- [ ] Wording avoids certified-safety claims.
- [ ] Wording does not imply wallet security, smart-contract auditing, or transaction simulation.
- [ ] Wording does not imply complete agent safety or full AI alignment.
- [ ] Limitations or non-claims are included where appropriate.

## PythiaLabs positioning checks

The PR should preserve this project boundary:

```text
PythiaLabs evaluates proposed AI-agent actions before execution and produces deterministic reviewer-facing evidence.
```

The PR should not reposition PythiaLabs as:

- a Web3 transaction simulator;
- a wallet-security product;
- a smart-contract audit tool;
- a generic memory product;
- a production enforcement platform;
- a certified compliance product.

## Evidence/reviewer docs checks

For reviewer-facing docs, verify that the PR links to relevant existing references when appropriate:

- `docs/REVIEWER_PATH.md`
- `docs/NON_CLAIMS.md`
- `docs/PORTFOLIO_RELATIONSHIP.md`
- `docs/evidence_artifact_schema.md`
- `docs/paid_review_demo_reviewer_checklist.md`
- `examples/paid_review_demo_expected_output.md`

## Safety checks for examples

If the PR adds examples:

- [ ] Examples are synthetic or sanitized.
- [ ] No real user data is included.
- [ ] No sensitive credentials, tokens, or secrets are included.
- [ ] No harmful operational instructions are introduced.
- [ ] Example digest/signature values are clearly placeholders unless generated by the demo.

## Validation expectations

For docs-only changes:

```text
No runtime validation required unless links, generated docs, site pages, or examples are changed.
```

For site/docs-index changes:

```bash
cd site
npm install
npm run build
```

For demo-related documentation changes, reviewers may request:

```bash
mix deps.get
mix test
make demo
```

## Approval rule of thumb

Approve when:

- the PR is narrow;
- links work;
- claims stay conservative;
- scope matches the issue;
- no runtime/site/workflow files were touched unexpectedly.

Request changes when:

- claims are inflated;
- scope has expanded;
- production/compliance/security claims are introduced;
- the PR changes runtime/site/workflow files without explicit issue scope;
- examples include sensitive or unsafe content.

## Bottom line

Docs PRs should make the project easier to review without making the project sound more mature than it is.
